# 仮ボルト注文明細 — 工区まとめ設定と表タイトル生成 専用引き継ぎ資料

> **この資料の目的**  
> モバイル側が仮ボルト注文明細の「工区別（詳細）モード」における工区まとめ設定と表タイトル粒度を推論実装しないための専用仕様書です。  
> 現状の問題: モバイルが「エリア名（`3工区, 4工区`）」という粗い粒度でタイトルを表示しているが、正しくは「フロア×工区の組み合わせ（`2F-3工区 + 3F-2工区`）」という詳細粒度でなければなりません。

---

## 1. 仮ボルト注文明細 工区別（詳細）モードの目的

「工区別（詳細）モード」は、**フロアと工区を両方含む最小粒度のロケーション単位**で仮ボルトを集計・表示するモードです。

「どの階のどの工区」で何本の仮ボルトが必要かが分かることが目的です。  
その上で、工区まとめ設定を使って「2F-1工区と3F-2工区を合算して発注する」という単位を自由に定義できます。

### 3つの表示モードとの位置づけ

| モード名 | ボタンラベル | 集計の入力単位 | 代表的なタイトル |
|---|---|---|---|
| **工区別（詳細）** | `工区別 (詳細)` | フロア×工区（`2-1`, `3-2`, `R-1`） | `No.1 (2-1, 3-2)` |
| 工区別（集計） | `工区別 (集計)` | エリア（`1工区`, `2工区`） | `No.1 (1工区, 2工区)` |
| フロア別（集計） | `フロア別 (集計)` | 階層（`2階`, `R階`） | `2階`, `3階` |

**「詳細」がつくのは、集計の入力単位がフロア×工区（=ロケーション）レベルだから**です。エリアや階層にまとめてしまうと「詳細」とは呼べません。

## 2. 仮ボルトの工区まとめ設定の意味

### Firestoreの保存先

```
projects/{projectId}.groupingSettings.tempBolt
```

### データ形式

```json
{
  "groupingSettings": {
    "tempBolt": {
      "2-1": 1,
      "2-2": 2,
      "2-3": 1,
      "3-1": 3,
      "3-2": 1,
      "R-1": 2
    }
  }
}
```

**キーはフロア×工区の生IDです。**  
simpleモード: `"{階数}-{工区番号}"` — 例: `"2-1"`, `"3-2"`, `"R-1"`, `"PH-1"`  
advancedモード: `"{customLevel}-{customArea}"` — 例: `"1F-A区"`, `"2F-B区"`

### 何をまとめる設定か

同じグループ番号（No.）を割り当てたロケーション（フロア×工区の組）が合算されます。

上の例では:
- `2-1`, `2-3`, `3-2` → No.1 に割り当て → 合算して1テーブルで表示
- `2-2`, `R-1` → No.2
- `3-1` → No.3

### 本ボルト用との違い

| | 本ボルト（honBolt） | 仮ボルト（tempBolt） |
|---|---|---|
| 保存フィールド | `groupingSettings.honBolt` | `groupingSettings.tempBolt` |
| キーの粒度 | フロア×工区（`"2-1"`） | フロア×工区（`"2-1"`） **同じ** |
| 設定の独立性 | 独立（仮ボルト設定と無関係） | 独立（本ボルト設定と無関係） |
| モードの種類 | 2種類（詳細/フロア別） | 3種類（詳細/集計/フロア別） |

キーの粒度は本ボルトと同じですが、**設定は完全に独立**しています。本ボルトの設定を変えても仮ボルトの設定は変わりません。

## 3. 表タイトルの生成ルール（最重要）

### 工区別（詳細）モードでの生成ロジック

Web版の `calculateLocalAggregation()` 関数が生成するタイトルの形式:

```
No.{グループ番号} ({ロケーションID1}, {ロケーションID2}, ...)
```

- 「`No.`」が必ずつく
- グループ番号は整数（1から始まる）
- 括弧の中に、そのグループに属するロケーションIDを **`, `（カンマ＋スペース）** で連結
- ロケーションIDは生の形式（`"2-1"`, `"R-1"` 等）

### simpleモードでの具体例

| 状況 | タイトル |
|---|---|
| `2-1` 単独（No.1） | `No.1 (2-1)` |
| `2-1` と `3-2` を合算（No.1） | `No.1 (2-1, 3-2)` |
| `2-1`, `2-3`, `R-1` を合算（No.1） | `No.1 (2-1, 2-3, R-1)` |
| `2-2` 単独（No.2） | `No.2 (2-2)` |

### advancedモードでの具体例

advancedモードではロケーションIDそのものにフロア名・エリア名が含まれているため、生のIDを表示しても人間が読める表示になります。

| 状況 | タイトル |
|---|---|
| `1F-A区` 単独（No.1） | `No.1 (1F-A区)` |
| `1F-A区` と `2F-B区` を合算（No.1） | `No.1 (1F-A区, 2F-B区)` |

### 連結記号の整理

| 用途 | 記号 | 例 |
|---|---|---|
| 仮ボルト詳細モードのタイトル内（ロケーション間） | `, `（カンマ＋スペース） | `No.1 (2-1, 3-2)` |
| 本ボルト詳細モードのテーブルタイトル | ` + `（スペース＋プラス＋スペース） | `2-1 + 3-2` |

**仮ボルトと本ボルトで記号が違います。** 仮ボルトはカンマ、本ボルトはプラスです。  
また、仮ボルトは「`No.X (...)`」形式で番号が先頭につきますが、本ボルトはつきません。

## 4. 本ボルトとの違い

### タイトル生成の比較

| 観点 | 本ボルト（工区別詳細） | 仮ボルト（工区別詳細） |
|---|---|---|
| タイトル形式 | `{locId1} + {locId2}` | `No.{N} ({locId1}, {locId2})` |
| 先頭に「No.X」 | **なし** | **あり** |
| 連結記号 | ` + ` | `, ` |
| 工区まとめUI | 折り畳みパネル内 | 折り畳みパネル内（同じ） |
| 工区まとめ設定の保存 | `groupingSettings.honBolt` | `groupingSettings.tempBolt` |
| データソースのキー | フロア×工区 (`"2-1"`) | フロア×工区 (`"2-1"`) |

### モバイルで「本ボルトと同じノリで実装してはいけない」点

1. **タイトル形式が異なる**: 本ボルトは `2-1 + 3-2`、仮ボルトは `No.1 (2-1, 3-2)` です。形式を統一しないでください。
2. **設定の保存フィールドが独立**: 両方を同じフィールドに書いてはいけません。
3. **仮ボルトには「工区別（集計）」という第3のモードがある**: 本ボルトにはこのモードがありません。詳細モードと集計モードを混同しないよう注意が必要です。

## 5. 表示粒度とタイトル粒度

### 粒度の階層

```
階層（フロア）
  └── ロケーション（フロア × 工区） ← 詳細モードの入力単位
        └── エリア（工区のみ、フロア横断）  ← 集計モードの入力単位
              └── 建物全体  ← フロア別集計モードが崩す粒度
```

### 各モードで「何が1テーブルになるか」

| モード | 工区まとめ前の最小単位 | 工区まとめ後の1テーブル |
|---|---|---|
| 工区別（詳細） | ロケーション（`"2-1"`, `"3-2"` など） | 同じNoのロケーション群（フロア横断可） |
| 工区別（集計） | エリア（`"1工区"`, `"2工区"` など） | 同じNoのエリア群 |
| フロア別（集計） | 階層（`"2階"`, `"3階"` など） | 工区まとめなし、階層がそのままタイトル |

### 表タイトルは何を表すのか（明確化）

工区別（詳細）モードの表タイトルは **「このグループに含まれる全ロケーション（フロア×工区）の列挙」** です。

- 「工区グループ名」（単なる番号付け）ではありません
- 「エリア単位の集約名」でもありません
- **「どのフロアのどの工区が合算されているかを列挙したもの」** です

タイトルを見ただけで「2階の1工区と3階の2工区を合算した結果」だと分かることが必要です。

## 6. モバイル向け再設計時の必須要件

### A. 詳細モードのデータソースはロケーションID単位でなければならない

工区別（詳細）モードでは、集計の入力データとして `filteredTempBolts`（キー: フロア×工区のロケーションID）を使います。エリア単位で集約した `aggregateTempBySection()` の結果（キー: `"1工区"`, `"2工区"`）を詳細モードに使うのは誤りです。

**禁止**: 詳細モードのデータソースにエリア名（`"1工区"`）レベルのキーを使うこと。

### B. 工区まとめ設定UIの各カードはロケーション単位

詳細モードの工区まとめ設定パネルには、ロケーション単位（`2-1`, `2-2`, `3-1`, `R-1` など）のカードが並びます。エリア単位（`1工区`, `2工区`）のカードが並ぶのは集計モードのUIです。

### C. タイトルはロケーションID（またはその人間可読変換）を含める

タイトルに使う情報はフロア×工区の組み合わせでなければなりません。simpleモードの生IDは `"2-1"` のような形式で人間には読みにくいため、モバイル側でヒューマナイズ（`"2-1"` → `"2F-1工区"` など）することが推奨されます。

**モバイル推奨タイトル形式（ヒューマナイズ版）**:
```
No.1 (2F-1工区, 3F-2工区)
```

または ` + ` 区切り（ユーザー要望に合わせて選択可）:
```
No.1 (2F-1工区 + 3F-2工区)
```

どちらの形式でも「フロア×工区」の情報が含まれていれば要件を満たします。

### D. 集計モードのタイトルと混同しない

| モード | タイトルに含まれるべき情報 |
|---|---|
| 工区別（詳細） | フロア + 工区（例: `2F-1工区`, `3F-2工区`） |
| 工区別（集計） | エリアのみ（例: `1工区`, `2工区`）、フロアは含めない |
| フロア別（集計） | 階層名のみ（例: `2階`, `3階`） |

## 7. 具体例

### 前提条件

- simpleモード、4階建て（2F, 3F, RF）、4工区
- 存在するロケーション（仮ボルトデータがあるもの）: `2-3`, `2-4`, `3-2`, `3-3`, `R-1`

### 工区まとめ設定（groupingSettings.tempBolt の内容）

```json
{
  "2-3": 1,
  "2-4": 2,
  "3-2": 1,
  "3-3": 3,
  "R-1": 2
}
```

### 工区別（詳細）モードでの表示

ロケーション `2-3` と `3-2` が No.1（同じグループ）、`2-4` と `R-1` が No.2、`3-3` が No.3 です。

#### Web版が生成するタイトル（raw ID形式）

```
No.1 (2-3, 3-2)     ← 2F-3工区 と 3F-2工区 を合算
No.2 (2-4, R-1)     ← 2F-4工区 と RF-1工区 を合算
No.3 (3-3)          ← 3F-3工区 単独
```

#### モバイル推奨タイトル（ヒューマナイズ版）

simpleモードの場合、`"{階数}-{工区}"` → `"{階数}F-{工区}工区"` に変換して表示することを推奨:

```
No.1 (2F-3工区 + 3F-2工区)
No.2 (2F-4工区 + RF-1工区)
No.3 (3F-3工区)
```

### 工区別（集計）モードでの表示（参考）

集計モードでは `aggregateTempBySection()` でエリア単位に変換してから集計します。

入力データのキーが `"3工区"`, `"4工区"` 等のエリア名に変換された後、工区まとめ設定の `tempBolt` フィールド（エリア名ベースに更新されている場合）が適用されます。

**このエリア名ベースのタイトルを詳細モードに使うと、どのフロアが含まれているかが分からなくなります。これが現在の問題です。**

## 8. 実装者向け注意事項

### 注意 1: 詳細モードと集計モードはデータソースが根本的に異なる

| | 詳細モード | 集計モード |
|---|---|---|
| データソース関数 | `filteredTempBolts`（そのまま） | `aggregateTempBySection()` |
| データのキー | `"2-1"`, `"3-2"` 等（フロア×工区） | `"1工区"`, `"2工区"` 等（エリアのみ） |
| 工区まとめ設定のキー | `"2-1"` 形式で保存・読み込み | `"1工区"` 形式で保存・読み込み |

モードが切り替わったとき、工区まとめ設定（`currentTempGroupingState`）もリセットする必要があります。Web版ではモード切替のたびに `currentTempGroupingState` を全削除して初期化します。

### 注意 2: simpleモードのロケーションIDはそのままでは人間が読みにくい

`"2-1"` は `2F-1工区`、`"R-1"` は `RF-1工区` を意味しますが、生のIDはその読み替えを要求します。モバイルでは表示時にヒューマナイズすることを強く推奨します。

変換ルール（simpleモード）:
```
"{f}-{s}" → "{f}F-{s}工区"  （例: "2-1" → "2F-1工区"）
"R-{s}"   → "RF-{s}工区"   （例: "R-1" → "RF-1工区"）
"PH-{s}"  → "PH-{s}工区"   （例: "PH-1" → "PH-1工区"）
```

advancedモードでは `"{customLevel}-{customArea}"` がそのままヒューマナイズ済みIDとして使えます。

### 注意 3: 表タイトルの連結記号

Web版は `, `（カンマ＋スペース）を使います。モバイルが ` + `（プラス）を使う場合、「本ボルト詳細モードのタイトルと同じ記号」になりますが、問題はありません。重要なのは「フロア×工区の情報が含まれている」ことです。

### 注意 4: 工区まとめ設定のグループ番号は「見た目の連番」ではない

設定値の `No.` 番号（1, 2, 3...）は `groupingSettings.tempBolt` に保存されている整数値です。ユーザーが「2工区をNo.5にした」後に他の工区を変更しても、No.5は残ります。表示時はタイトル内の `,` 区切りロケーション一覧を昇順で並べるだけで、グループ番号の連続性は保証されません。

### 注意 5: `groupingSettings.tempBolt` のキー形式はモードに依存しない

`groupingSettings.tempBolt` は常にフロア×工区のロケーションID（`"2-1"`, `"R-1"` 等）をキーとして保存します。集計モードでは `aggregateTempBySection()` で変換した後に別途グループ管理します。Firestoreの `groupingSettings.tempBolt` は詳細モード専用の設定です。

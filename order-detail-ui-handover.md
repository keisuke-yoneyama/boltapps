# 注文明細UI 専用引き継ぎ資料

> **この資料の目的**  
> モバイル側が注文明細UIを「推測で実装しないための」専用仕様書です。  
> 現状のモバイル版で発生している「工区まとめUIの未実装」「柱用ボルトの混在」という2点のズレを解消することを優先課題としています。

---

## 1. 注文明細UIの目的

注文明細は「発注書として使える集計表」を生成する画面です。目的は次の1点に尽きます。

**「この工事で、どのサイズのボルトを何本発注すればよいか」を工区グループ単位で把握すること。**

タリーシートや本数表（フロア工区別クロス集計）とは根本的に異なります。タリーシートは入力帳票、本数表は生データの確認表です。注文明細は「実際に発注する単位で合算した最終出力」です。

---

## 2. 注文明細UIの基本構造

注文明細全体は5つのセクションで構成されています。セクションは固定順で描画されます。

```
注文明細 全体
│
├─ [1] 本ボルト注文明細
│     ├─ ヘッダー（タイトル + 表示モード切替ボタン）
│     ├─ 工区まとめ設定パネル（折り畳み式、詳細モード時のみ表示）
│     └─ テーブル群（グループごとに1枚、最後に「柱用」テーブル1枚）
│
├─ [2] 工場地組み用 注文明細
│     └─ テーブル1枚（全工区合算済み）
│
├─ [3] 地組み用 注文明細
│     └─ テーブル1枚（全工区合算済み）
│
├─ [4] D-Lock 注文明細
│     └─ テーブル1枚（全工区合算済み）
│
└─ [5] 中ボルト注文明細
      └─ テーブル1枚（全工区合算済み）
```

> **重要**: セクション[2]〜[5]は工区別集計ではありません。「全工区の合算値を1テーブルで表示」するだけです。工区まとめ設定の影響を受けるのは **セクション[1]のみ** です。

---

## 3. 工区別モードの表示構造

セクション[1]（本ボルト注文明細）の詳細モード時の画面構成を説明します。

### 画面レイアウト（詳細モード）

```
┌───────────────────────────────────────────┐
│ ■ 本ボルト注文明細                         │  ← セクションヘッダー
│   S10T / F8T / 柱用ボルト                 │
│                      [工区別(詳細)] [フロア別(集計)] │  ← モード切替
├───────────────────────────────────────────┤
│ ▼ 工区まとめ設定                           │  ← 折り畳みパネル（詳細モードのみ）
│   詳細モード用：同じ番号を選択した工区を合算します  │
│   [2-1: No.1▼] [2-2: No.2▼] [2-3: No.2▼] [リセット] │
├───────────────────────────────────────────┤
│                                           │
│  [テーブル: "2-1"]                        │  ← グループ1（単独の場合）
│  [テーブル: "2-2 + 2-3"]                 │  ← グループ2（2工区まとめた場合）
│  [テーブル: "柱用"]                       │  ← 全工区の柱ボルト合算（紫色）
│                                           │
└───────────────────────────────────────────┘
```

### テーブルのタイトル名の形式

テーブルのタイトル（グループ名）は、工区まとめ設定の結果によって自動生成されます。

| 状況 | タイトル例 |
|---|---|
| 単独工区（まとめなし） | `2-1`、`2-2`、`R-1` |
| 2工区をまとめた場合 | `2-1 + 2-2` |
| 3工区をまとめた場合 | `2-1 + 2-2 + 2-3` |

> **注意**: タイトルは「No.X (...)」の形式ではありません。含まれる工区IDを ` + ` でつないだだけの文字列です。「No.X」形式を使うのは仮ボルト注文明細のみです（後述）。

## 4. 工区まとめ機能のUIと操作仕様

### UIの実体

「工区まとめ設定」は `<details>` 要素（折り畳みパネル）として実装されています。

- **表示条件**: 詳細モード（「工区別 (詳細)」選択時）のみ表示。フロア別モード時は非表示。
- **場所**: セクションヘッダーの直下、テーブル群の直前。

### パネル内の構成

```
┌─────────────────────────────────────────┐
│ ▼ 工区まとめ設定                         │  ← クリックで開閉
│   詳細モード用：同じ番号を選択した工区を合算します │
├─────────────────────────────────────────┤
│  [リセット]                         ←右上ボタン│
│                                         │
│  ┌────────┐  ┌────────┐  ┌────────┐   │
│  │  2-1   │  │  2-2   │  │  2-3   │   │
│  │[No.1▼] │  │[No.2▼] │  │[No.2▼] │   │
│  └────────┘  └────────┘  └────────┘   │
│  ┌────────┐  ┌────────┐               │
│  │  3-1   │  │  R-1   │               │
│  │[No.3▼] │  │[No.4▼] │               │
│  └────────┘  └────────┘               │
└─────────────────────────────────────────┘
```

### 操作の意味

1. **各カードのラベル**: 工区ID（例: `2-1`, `2-2`, `R-1`）。simpleモードでは `"{階数}-{工区番号}"` 形式。
2. **各カードのドロップダウン**: `No.1` 〜 `No.N`（N = 工区の総数）。同じ番号を選ぶと合算される。
3. **リセットボタン**: 全工区をそれぞれ別々の番号（No.1, No.2, No.3...）に戻す。

### 操作例

工区が `2-1`, `2-2`, `2-3`, `3-1`, `R-1` の5つある場合：

| 工区 | 選択したNo. | 結果 |
|---|---|---|
| 2-1 | No.1 | グループ1に属する |
| 2-2 | No.1 | グループ1に合算される |
| 2-3 | No.2 | グループ2（単独） |
| 3-1 | No.3 | グループ3（単独） |
| R-1 | No.3 | グループ3に合算される |

→ 生成されるテーブル:
- `2-1 + 2-2` の集計テーブル
- `2-3` の集計テーブル
- `3-1 + R-1` の集計テーブル
- `柱用` の合算テーブル（全工区分）

### ドロップダウンの選択肢の上限

選択肢の上限（最大No.値）は「**工区の総数**」です。5工区なら No.1〜No.5 が選択肢に並びます。これは「全部バラバラ」が最多グループ数だからです。

### 工区まとめを変更したとき

ドロップダウンを変更すると即時に集計が再計算され、テーブルが再描画されます。変更後600msのdebounceでFirestoreに保存されます（ユーザーが気づかない自動保存）。

## 5. 工区まとめ設定の保存先と継承方法

### 保存先

Firestoreの以下のパスに保存されています。

```
artifacts/{appId}/public/data/projects/{projectId}
  .groupingSettings.honBolt   ← 本ボルト用の工区まとめ設定
  .groupingSettings.tempBolt  ← 仮ボルト用の工区まとめ設定
```

### データ形式

```json
{
  "groupingSettings": {
    "honBolt": {
      "2-1": 1,
      "2-2": 1,
      "2-3": 2,
      "3-1": 3,
      "R-1": 3
    },
    "tempBolt": {
      "1工区": 1,
      "2工区": 1,
      "3工区": 2
    }
  }
}
```

キー: 工区ID文字列  
値: グループ番号（1以上の整数）

### 本ボルトと仮ボルトの設定が独立している理由

本ボルトと仮ボルトでは「集計の単位（キーの形式）」が異なります。

- 本ボルトの詳細モード: キーは `"2-1"`, `"2-2"` などの `"{階}-{工区}"` 形式
- 仮ボルトの詳細モード: キーも `"2-1"` 形式（同じ）
- 仮ボルトの集計モード: キーは `"1工区"`, `"2工区"` などのエリア名形式

そのため、同じフィールドに保存できず独立しています。

### 読み込みタイミングと復元ロジック

プロジェクトを開いて注文明細を表示する際、初回（または工区構成が変化した後）に保存済み設定を復元します。

**復元条件**: 現在の工区IDリストと保存済みキーリストを照合し、「新しい工区が存在する（= 保存済みキーに含まれていない工区がある）」場合に全リセットして再読み込みします。

```
保存済み設定があり、現在の工区と照合して一致する場合 → 保存済み設定を復元
保存済み設定がない、または工区構成が変わった場合 → 全工区を別々（デフォルト）に初期化
```

### モバイルでの継承方針

モバイルが読み取り専用であっても、Firestoreから `groupingSettings` を読み込んで、Web版で設定された工区まとめ状態を引き継いで表示することが推奨です。

**モバイルで工区まとめ設定を変更・保存する場合**:  
同じフィールド（`groupingSettings.honBolt`, `groupingSettings.tempBolt`）に書き込めば、Web版とリアルタイムに設定が共有されます。Firestoreのdot notation (`updateDoc` + 部分更新) を使うことで、本ボルト設定を変えても仮ボルト設定は壊れません。

## 6. 柱用ボルトの独立集計表示ルール

これはモバイル版で最も重要な修正点です。

### 現状の問題

モバイル版では「柱用ボルト（`M22×60(本柱)` のようにサフィックスが付いたもの）」が通常の注文明細テーブルに混在しています。これは誤りです。

### 正しい挙動（Web版の実装）

#### Step 1: 計算段階でサフィックスが付く

`calculateResults()` では、柱用ボルトのサイズ文字列に `(本柱)` サフィックスを付けます。

```
M22×60     → 通常ボルト → そのまま "M22×60"
M22×60     → 柱継手 or isBundledWithColumn=true → "M22×60(本柱)"
```

**対象**: `joint.type === "column"` または `joint.isBundledWithColumn === true` の継手。

#### Step 2: 描画段階でサフィックスで振り分ける

`renderOrderDetails()` の中でサイズ文字列を走査し、`(本柱)` が含まれるものを **専用バケツ（`specialBolts.column`）** に分離します。

```js
// 各工区、各サイズを走査する
if (size.includes("(本柱)")) {
  // 専用バケツへ（全工区分をマージ）
  mergeSpecial(specialBolts.column, size, data);
} else {
  // 通常テーブル用データへ（工区まとめが適用される）
  filteredHonBolts[locId][size] = data;
}
```

#### Step 3: 柱用は「全工区合算の1テーブル」として末尾に表示

通常ボルトが工区まとめ設定によって複数のグループテーブルになるのに対し、柱用ボルトは **全工区をまとめた1枚のテーブル** として表示されます。

テーブルタイトル: `柱用`  
テーブル色: 紫

表示順序:
```
[グループ1のテーブル]  ← 工区まとめに従ったグループ
[グループ2のテーブル]
[グループ3のテーブル]
[柱用テーブル]         ← 全工区合算、工区まとめとは無関係
```

#### Step 4: 表示時に `(本柱)` サフィックスを除去

柱用テーブルの行表示時は、ボルトサイズ文字列から `(本柱)` を取り除いて表示します。

```
内部キー: "M22×60(本柱)"
表示文字列: "M22×60"   ← (本柱) を除いて表示
```

これはテーブルヘッダーがすでに「柱用」という見出しを持っているため、行にサフィックスは不要です。

### 参考表（フロア工区別本数表）との違い

参考表（`renderResults()` の `floorTableHtml` 部分）では、ボルトサイズをそのまま行として表示しています。`M22×60(本柱)` という行と `M22×60` という行が独立して存在します。参考表は「生データの確認用」なので、このまま分類なしで出しています。

| | 注文明細（セクション[1]） | 参考表（フロア工区別） |
|---|---|---|
| 柱用ボルト | 専用テーブルとして分離・末尾表示 | `M22×60(本柱)` 行として他と混在 |
| 工区まとめ | 適用される | 適用されない（生データ） |
| 重量表示 | あり | なし |

### モバイルでの実装指針

1. Firestoreのデータを受け取ったら、注文明細描画時に必ずサイズ文字列を走査する
2. `size.includes("(本柱)")` → 柱用バケツへ
3. `size.includes("(工場地組)")` → 工場地組みバケツへ
4. `size.includes("(地組)")` → 地組みバケツへ
5. `size.startsWith("D")` → D-Lockバケツへ
6. `size.startsWith("中ボ")` → 中ボルト(Mネジ)バケツへ
7. `size.startsWith("中")` → 中ボルト(ミリ)バケツへ
8. それ以外 → 通常ボルト（工区まとめ適用対象）へ

この振り分けをしないと、通常テーブルに `M22×60(本柱)` が混在します。

## 7. 参考表との役割分離

注文明細UIと参考表は全く異なる目的を持っています。

| 観点 | 注文明細 | 参考表（本ボルト本数表） |
|---|---|---|
| 目的 | 発注書の元データ | 工区別の生データ確認 |
| 集計単位 | 工区まとめ設定後のグループ | 各工区（まとめなし） |
| 柱用ボルト | 独立テーブルで分離表示 | 他サイズと同行（サフィックスあり） |
| 重量表示 | あり | なし |
| 工区まとめ設定 | 影響する | 影響しない |
| テーブル構造 | グループ1枚ずつのカード型 | 全工区を列にした横長クロス表 |

### 「参考表では柱用が正しく見えるが注文明細がズレている」理由

参考表は `M22×60(本柱)` という行をそのまま表示しているだけなので、「柱用が分かれているように見える」のです。これは分離されているわけではなく、単に行キーが違うだけです。

注文明細では、この行を「柱用バケツに移して通常テーブルから除外する」処理が追加で必要です。参考表にはその処理がありません。

## 8. 仮ボルト注文明細の固有仕様

仮ボルト注文明細は本ボルトとほぼ同じ構造ですが、以下の点が異なります。

### 表示モードが3つある（本ボルトは2つ）

| モード名 | ボタンラベル | 集計単位 | 工区まとめ設定の使用 |
|---|---|---|---|
| 工区別（詳細）| `工区別 (詳細)` | `2-1`, `2-2` などの個別工区 | 使う |
| 工区別（集計）| `工区別 (集計)` | `1工区`, `2工区` などのエリア | 使う（エリア単位で） |
| フロア別（集計）| `フロア別 (集計)` | `2階`, `3階` などの階層 | 使わない |

### グループキーの形式が異なる

本ボルト:
```
"2-1 + 2-3"    ← 工区IDを " + " でつないだだけ
```

仮ボルト:
```
"No.1 (1工区, 2工区)"    ← "No.{番号} ({エリア名, ...})" 形式
"No.2 (3工区)"
```

仮ボルトでは「No.X」形式を使います。本ボルトでは使いません。

### 工区別（集計）モードとは

仮ボルトの中間集計モードです。工区まとめ設定を適用しますが、集計単位がフロアをまたいだ「エリア」です。

```
simpleモード例: 工区が 2-1, 2-2, 3-1, 3-2, R-1, R-2 のとき
  "1工区" = 2-1 + 3-1 + R-1 を合算したもの
  "2工区" = 2-2 + 3-2 + R-2 を合算したもの
```

これをさらに工区まとめ設定で合算できます。

### 仮ボルトには柱用・地組み用・D-Lock・中ボルトの分離がない

仮ボルトは通常ボルト1種類のみです。`(本柱)`, `(地組)` などのサフィックスは付きません。また、重量列もありません。

## 9. モバイル向け再設計時の必須要件

以下は「削ってはいけない概念」です。UIを再設計しても、これらの機能・挙動は維持してください。

### A. 工区まとめ設定は操作可能にする

ユーザーが任意の工区を任意のグループ番号に変更できるUIが必要です。Web版の「ドロップダウン方式」をそのままスマホに持ち込む必要はありませんが、以下の概念は必須です:

- 各工区に「グループ番号」を割り当てられること
- 同じグループ番号を持つ工区が合算されること
- 設定をリセット（全工区バラバラ）できること
- 設定がFirestoreに保存されること（変更後に再度開いても維持される）

### B. 工区まとめ設定はWeb版の設定を継承する

モバイルで初回表示するとき、Firestoreの `groupingSettings.honBolt` / `groupingSettings.tempBolt` を読み込んで反映させてください。Web版でまとめた設定がモバイルでも同じように見える状態が正しい。

### C. 柱用ボルトは独立セクションで表示する

通常の注文明細テーブルとは別の「柱用」テーブルとして分離して表示してください。サイズ文字列で `(本柱)` が含まれているかどうかで判定します。表示時にはサフィックスを除去してください。

### D. グループ単位と行単位を崩さない

- テーブル（= グループ）: 工区まとめ後の1集計単位
- 行: 1ボルトサイズ

この2階層は維持してください。「全サイズを1リストにフラット化する」のは誤りです。工区Aと工区Bでは同じサイズのボルト本数が異なります。グループを分けることに意味があります。

### E. 内訳（部材ごとの本数）は表示できるようにする

各セル（工区グループ × ボルトサイズ）をタップすると「どの部材が何本使っているか」を確認できる機能が必要です。この内訳データはFirestore側では計算されておらず、アプリ側で計算するときに `joints` マップ（部材名→本数）として保持されます。

## 10. 実装者向け注意事項

### 注意 1: Firestoreに注文明細の計算結果は保存されていない

`resultsByLocation`（位置別ボルト本数）や `aggregatedCounts`（グループ合算後）はFirestoreに保存されていません。プロジェクトデータ（`tally`, `joints`, `members`）から毎回計算します。

### 注意 2: 工区まとめ設定は「UI状態」ではなくDBに保存されている

工区まとめ設定は一時的なUIフィルターではなく、Firestoreに永続保存されています。モバイルで変更する場合は `updateDoc` で部分更新してください。

```
更新パス: groupingSettings.honBolt  （ドット記法）
書き込む値: { "2-1": 1, "2-2": 1, "2-3": 2, ... }  （全工区分のオブジェクト）
```

### 注意 3: 本ボルトのグループキーは "No.X" 形式ではない

「No.1 (2-1 + 2-2)」のような形式は仮ボルト専用です。本ボルトのグループタイトルは単に「`2-1 + 2-2`」（工区IDを ` + ` でつなぐ）です。混同しないようにしてください。

### 注意 4: 柱用テーブルは工区まとめを適用しない

柱用ボルトは全工区の値を無条件で1テーブルに合算します。工区まとめ設定の影響を受けません。設定が変わっても柱用テーブルの値は変わりません。

### 注意 5: フロア別モードのとき工区まとめ設定UIを非表示にする

フロア別モードでは工区まとめ設定は無意味（使わない）なので、工区まとめ設定パネルを非表示にします。「ある設定が存在するのに使われない」と混乱を招くためです。

### 注意 6: advancedモードでは工区IDの形式が違う

`simple`モード（`mode !== "advanced"`）: `"2-1"`, `"3-2"`, `"R-1"` 形式  
`advanced`モード: `"{customLevel}-{customArea}"` 形式（例: `"1F-A区"`, `"2F-B区"`）

工区まとめ設定のキー形式もこれに合わせて異なります。

### 注意 7: 工区まとめ設定の初期化タイミング

以下の場合に設定を初期化してください:

| 条件 | 処理 |
|---|---|
| 初回表示（設定なし） | 全工区をそれぞれ別グループに初期化（No.1, No.2, No.3...） |
| 設定あり、工区構成が一致 | 保存済み設定を復元 |
| 設定あり、工区構成が変わった | 設定をリセットし、全工区別グループに初期化 |

「工区構成が変わった」の判定: 現在の工区キーリストのいずれかが保存済み設定のキーに存在しない場合。
